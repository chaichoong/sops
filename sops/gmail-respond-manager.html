🎯 1. Purpose & Scope

Manages the full 'To Respond' email workflow for kevin@runpreneur.org.uk. Fetches all emails labelled '1: to respond', drafts replies using the knowledge base, presents all drafts for batch approval, creates Gmail drafts, and guides label management after sending. Self-improving: knowledge base is updated after each session when Kevin amends a draft.

ℹ️
Scope: In scope: emails labelled '1: to respond' in kevin@runpreneur.org.uk. Out of scope: unlabelled emails, emails in other accounts, or sending emails directly (Claude creates drafts; Kevin sends from Gmail).

💡 2. When to Use This Skill

💬 Trigger Phrases

Use this skill when Kevin or the team says any of the following:

  • "Process my To Respond emails"
  • "Draft my email replies"
  • "Work through my To Respond inbox"
  • "Handle my emails"
  • "Respond to my labelled emails"
⚠️
Do not use this skill when: Emails not in the '1: to respond' label — this skill only processes that specific label.

1

Fetch and Filter Emails

automated
  1. Search gmail: has:userlabels newer_than:90d
  2. Filter client-side: keep only threads where labelIds contains Label_1.
  3. For each thread: call gmail_read_thread (not just gmail_read_message) for full context.
  4. List emails found before proceeding. Stop if zero found.
ℹ️
Known issue: label:'1: to respond' search syntax does NOT reliably return results. Always use has:userlabels + client-side filter.
2

Draft Responses

automated
  1. Read both knowledge base files: tone-and-style.md and standard-responses.md.
  2. Draft each reply using full thread context, tone rules, and known patterns.
  3. Use [PLACEHOLDER: ...] markers for anything requiring Kevin's input (dates, figures, decisions).
  4. Flag spam/circulars as [SKIP — no reply needed] with a brief reason.
  5. ARP (Anglia Revenues Partnership) emails: do NOT draft replies to counciltax@angliarevenues.gov.uk — direct Kevin to their portal instead.
3

Batch Presentation and Approval

approval gate
  1. Present ALL drafts in a single structured block: sender, subject, received date, draft reply, confidence level, suggested next label.
  2. Ask: 'All approved' / '[N] amend: [change]' / '[N] skip' / '[N] label: Responded or Follow Up'.
  3. Apply all amendments. Update knowledge base if any amendment reveals a reusable pattern.
  4. Confirm final list before creating Gmail drafts. Wait for explicit 'confirm' before proceeding.
4

Create Gmail Drafts and Label Management

automated
  1. Call gmail_create_draft for each approved reply with correct to, subject (Re: original), body, and threadId.
  2. STOP after draft creation. Wait for Kevin to confirm emails are sent from Gmail UI.
  3. After send confirmation: provide label change list (Responded / Follow Up). Kevin applies labels manually — label automation is not functional in this environment.

🚨 8. Error Handling

  • Label search returns zero results: Switch to has:userlabels newer_than:90d + client-side filter.
  • gmail_create_draft fails: Report failure, continue with remaining drafts.
  • Sender email missing: Flag before creating draft — never leave the 'to' field blank.
  • Chrome label automation fails: This is expected — provide Kevin with a numbered list for manual labelling.

🛠️ 9. Troubleshooting

❌ Gmail label search returns no results
✅ Use has:userlabels newer_than:90d and filter client-side by labelIds containing Label_1.
❌ Draft created but sent to wrong address
✅ Always populate the 'to' field with the sender's actual email from the From header. Never guess.
