🎯 1. Purpose & Scope

Provides the authoritative guide for creating and updating Claude Cowork skills. Skills are modular, self-contained packages — a SKILL.md file plus optional bundled resources — that give the agent specialised knowledge and workflows it cannot fully possess from training alone.

This skill must be read before editing any existing skill file. It defines the 6-step creation process: understand, plan, initialise, edit, deliver, and iterate.

ℹ️
Scope: In scope: creating new SKILL.md files, updating existing skills, adding bundled resources (scripts, references, templates), validating and delivering skills. Out of scope: generating SOP documentation for a skill (use sop-generator), deploying skills to production infrastructure, or editing files directly without following this process.

💡 2. When to Use This Skill

💬 Trigger Phrases

Use this skill when Kevin or the team says any of the following:

  • "Create a new skill"
  • "Update the [skill] skill" / "Improve [skill]"
  • "Add a skill for [workflow]"
  • "Modify [skill name]" / "Fix [skill name]"
  • Any modification or improvement request to an existing skill
⚠️
Do not use this skill when: Kevin wants to generate the SOP documentation for a skill — use sop-generator for that. This skill creates and modifies the skill itself. Also: never edit skill files directly without reading this skill first.

1

Understand the Skill with Concrete Examples

manual
  1. Ask: "What functionality should this skill support?" and "Can you give examples of how it would be used?"
  2. Avoid asking too many questions at once. Conclude when you have a clear sense of the functionality.
  3. Skip this step only when usage patterns are already clearly understood from the request.
2

Plan the Reusable Skill Contents

manual
  1. For each usage example, identify reusable resources: scripts/ (code rewritten repeatedly), templates/ (same boilerplate each time), references/ (documentation needed repeatedly).
  2. Match specificity to task fragility: high freedom (text) for context-dependent tasks; medium freedom (pseudocode) when a preferred pattern exists; low freedom (specific scripts) for fragile/sequential operations.
  3. Plan what stays in SKILL.md vs what moves to reference files. Keep SKILL.md under 500 lines.
ℹ️
Avoid duplication — information lives in SKILL.md OR references, never both. Do NOT include README.md, CHANGELOG.md, or other auxiliary docs. Skills are for AI agents, not users.
3

Initialise the Skill

automated
  1. For new skills, run: python /home/ubuntu/skills/skill-creator-from-manus/scripts/init_skill.py <skill-name>
  2. The script creates the skill directory at /home/ubuntu/skills/<skill-name>/, generates a SKILL.md template with proper frontmatter and TODO placeholders, and creates example resource directories (scripts/, references/, templates/).
  3. Customise or remove the generated files as needed after initialisation.
  4. Skip this step only if the skill already exists and iteration is needed — go directly to Step 4.
4

Edit the Skill

manual
  1. Consult design pattern references as needed: references/workflows.md for multi-step processes, references/output-patterns.md for output formats, references/progressive-disclosure-patterns.md for splitting content across files.
  2. Build bundled resources first (scripts/, references/, templates/). Test scripts by running them. Delete any unused example files from initialisation.
  3. Write SKILL.md frontmatter: name (skill identifier) and description (primary trigger — must state what the skill does AND when to use it).
  4. Write SKILL.md body in imperative/infinitive form. Only include context the agent doesn't already have — challenge each paragraph for token cost.
  5. Apply three-level progressive disclosure: metadata (~100 words, always in context) → SKILL.md body (loads when triggered) → bundled resources (loaded as needed).
⚠️
The description frontmatter field is the only thing the agent reads to decide when to trigger the skill. Make it clear and comprehensive — the body only loads after triggering.
5

Validate and Deliver the Skill

automated
  1. Run: python /home/ubuntu/skills/skill-creator-from-manus/scripts/quick_validate.py <skill-name>
  2. Fix any validation errors and re-run until clean.
  3. Deliver via the message tool, sending the SKILL.md file path: /home/ubuntu/skills/{skill-name}/SKILL.md
  4. The system auto-detects the path pattern, packages the skill directory into a .skill file, and sends it to the frontend with options: Add to My Skills, Download, Preview.
6

Iterate Based on Real Usage

conditional
  1. Use the skill on real tasks and note any struggles or inefficiencies.
  2. Identify how SKILL.md or bundled resources should be updated.
  3. Return to Step 4, implement changes, re-validate, and re-deliver.
💡
Iteration often happens right after first use, while context of how the skill performed is still fresh.

🚨 8. Error Handling

  • Validation fails: Read the error output from quick_validate.py, fix the identified issues, and re-run until clean before delivering.
  • init_skill.py not found: Confirm the path /home/ubuntu/skills/skill-creator-from-manus/scripts/init_skill.py exists. If missing, create the directory structure manually.
  • Skill already exists: Skip Step 3. Read the existing SKILL.md first, then apply changes in Step 4.
  • SKILL.md exceeds 500 lines: Move variant-specific content to references/ files and replace with a pointer in SKILL.md describing when to read each reference.

🛠️ 9. Troubleshooting

❌ Skill not triggering in conversation
✅ The description frontmatter is the only field used for trigger matching. Rewrite it to clearly state what the skill does AND when to use it. The body is not read until after triggering.
❌ Skill too verbose — consuming too much context window
✅ Challenge every paragraph: does the agent really need this? Move reference content to references/ files. Keep SKILL.md under 500 lines. Prefer concise examples over verbose explanations.
❌ Script in scripts/ not working after init
✅ Test scripts by running them immediately after writing. For multiple similar scripts, test a representative sample. Delete any unused example files from initialisation.
❌ Existing skill edited directly without following this process
✅ Always read this skill before making any modification. Review changes against the Step 4 principles and validate with quick_validate.py before re-delivering.
